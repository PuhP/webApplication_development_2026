\subsubsection{Описание ролей и сценариев взаимодействия Студент--Преподаватель--Администратор}

На основе архитектурной схемы приложения выделяются три ключевые роли, взаимодействие которых 
обеспечивает замкнутый цикл управления учебным процессом.

\textbf{1. Роль <<Студент>>}

Основной сценарий взаимодействия студента с системой начинается с \textbf{авторизации} и 
выбора конкретной дисциплины.

\begin{itemize}
    \item \textbf{Запись:} Студент отправляет запрос на сервер (<<Запись в конец очереди>>), 
	после чего его данные попадают в таблицу текущей очереди.
    \item \textbf{Подтверждение (Check-in):} За 24 часа до начала занятия срабатывает логика 
	<<Таблицы готовности>>. Студент обязан нажать кнопку подтверждения. В случае отсутствия 
	действия сервер автоматически удаляет запись, реализуя механизм самоочистки очереди.
    \item \textbf{Ожидание:} Студент получает актуальное состояние очереди по протоколу WebSocket, 
	что исключает необходимость ручного обновления страницы.
\end{itemize}

\textbf{2. Роль <<Преподаватель>>}

Преподаватель взаимодействует с наиболее сложным интерфейсом, интегрированным с блоками 
\textbf{<<Таблица успеваемости>>} и \textbf{<<Таблица посещаемости>>}.

\begin{itemize}
    \item \textbf{Управление потоком:} Преподаватель инициирует вызов студента. Сервер 
	обновляет статус записи, и через WebSocket уведомление рассылается всем участникам.
    \item \textbf{Фиксация результата:} После защиты лабораторной работы преподаватель через форму 
	редактирования отправляет данные на сервер. Система одновременно обновляет <<Журнал>> (выставляя оценку) 
	и удаляет студента из текущей активной очереди, освобождая место для следующего.
\end{itemize}

\textbf{3. Роль <<Администратор (Староста)>>}

Данная роль является расширением роли студента и обладает административными правами в рамках своей учебной группы.

\begin{itemize}
    \item \textbf{Конфигурация:} Староста отвечает за создание структуры группы (добавление студентов в 
	<<Таблицу студентов>>) и привязку предметов.
    \item \textbf{Модерация:} В сценарии <<Редактирование очереди>> администратор может вручную изменять 
	порядок записей (например, если студенты договорились о взаимозамене), что сразу отражается в базе 
	данных и на клиентах других пользователей.
    \item \textbf{Контроль посещаемости:} Администратор имеет право выставлять первичные отметки о присутствии, 
	которые затем верифицируются преподавателем.
\end{itemize}

\textbf{Логика взаимодействия (Summary):}
Центральным узлом системы выступает \textbf{Сервер}, который координирует запросы от всех типов клиентов.
 Благодаря разделению на таблицы (предметы, студенты, группы, успеваемость) обеспечивается целостность данных: 
 оценка не может быть выставлена студенту, которого нет в базе группы, а запись в очередь невозможна без выбора 
 существующего предмета.

\paragraph{Пояснение по логике подтверждения:}
В таблице \texttt{queue\_entries} поле \texttt{is\_confirmed} по умолчанию имеет значение \texttt{false}. 
Фоновый процесс (task) на сервере или логика контроллера проверяет это поле за определенное время до начала
 занятия. Если к моменту «отсечки» значение остается \texttt{false}, запись переходит в статус 
 \texttt{cancelled} или удаляется физически, что мгновенно отражается у всех пользователей через WebSockets.

\subsubsection{Разработка макета Wireframe для Преподавателя и администратора}

Интерфейс для ролей с расширенными правами ориентирован на управление и быстрый ввод данных. 
В отличие от студенческого макета, здесь добавлены элементы управления каждой строкой очереди.

\textbf{Основные элементы макета:}

\begin{enumerate}
    \item \textbf{Панель управления сессией:} Кнопки «Начать прием», «Пауза», «Завершить».
    \item \textbf{Интерактивная очередь:} Каждая строка содержит кнопки «Вызвать», «Удалить», «Передвинуть».
    \item \textbf{Форма быстрого ввода (Modal):} При нажатии на студента открывается окно для 
	выставления оценки за конкретную лабораторную работу.
    \item \textbf{Вкладка «Журнал»:} Переключатель на просмотр общей ведомости группы 241-326.
\end{enumerate}

\textbf{Технологическое обоснование дизайна:}

\begin{itemize}
    \item \textbf{Сетка (Grid):} Для верстки будет использоваться CSS Flexbox, что позволит макетам 
	одинаково корректно отображаться как на десктопе (у преподавателя), так и на узких экранах смартфонов.
    \item \textbf{Цветовая индикация:} Статусы (Сдает, Ожидает, Подтвержден) будут иметь цветовую кодировку 
	(зеленый, желтый, синий), что позволит пользователю мгновенно считывать ситуацию без чтения текста.
    \item \textbf{WebSockets:} При нажатии преподавателем кнопки «Вызвать», у студента на макете мгновенно
	 обновится статус, а позиция «ВЫ» переместится в блок «ТЕКУЩИЙ».
\end{itemize}
