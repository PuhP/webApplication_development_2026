\subsubsection{Реализация модуля авторизации для администратора, разграничения прав доступа для всех ролей}

Для администратора создан отдельный маршрут регистрации с проверкой кода \texttt{ADMIN\_INVITE\_CODE}. Разграничение доступа на уровне маршрутов обеспечивается декоратором \texttt{@role\_required} (листинг~\ref{lst:role_decorator}), который принимает список допустимых ролей.

\begin{lstlisting}[caption={Декоратор role\_required (security.py)}, label={lst:role_decorator}]
def role_required(*roles: str):
    def decorator(view):
        @wraps(view)
        def wrapped(*args, **kwargs):
            if not current_user.is_authenticated or current_user.role not in roles:
                abort(403)
            return view(*args, **kwargs)
        return wrapped
    return decorator
\end{lstlisting}

Пример использования: \texttt{@role\_required("teacher", "admin")} перед маршрутами управления очередями и журналом. Для администратора доступны дополнительные маршруты: \texttt{/admin/groups}, \texttt{/admin/users}. В шаблонах проверка роли осуществляется через \texttt{current\_user.role} для условного отображения элементов интерфейса.

\subsection{Разработка CRUD интерфейса}

Реализованы полноценные CRUD-операции над основными сущностями: очереди, группы, пользователи, дисциплины. Взаимодействие клиент-сервер построено на HTTP-методах POST (создание, обновление) и GET (чтение), а также POST-запросах на удаление/изменение статуса.

\subsubsection{Разработка CRUD взаимодействия ``Клиент-Сервер'' для очередей}

Управление очередями осуществляется через маршруты, описанные в \texttt{main.py}. Основные операции:

\begin{itemize}
    \item \textbf{Создание очереди}: POST \texttt{/teacher/sessions} (с параметром \texttt{action=session}) — добавляет запись в таблицу \texttt{QueueSession}.
    \item \textbf{Запись студента}: POST \texttt{/sessions/<id>/join} — создаёт запись \texttt{QueueEntry}, вычисляет позицию через функцию \texttt{next\_position} (листинг~\ref{lst:next_pos}).
    \item \textbf{Вызов следующего}: POST \texttt{/sessions/<id>/call-next} — находит первый элемент в очереди со статусом \texttt{waiting} и переводит его в \texttt{checking} (функция \texttt{call\_entry}).
    \item \textbf{Перемещение записи}: POST \texttt{/entries/<id>/move/<direction>} — меняет позиции двух соседних записей.
    \item \textbf{Завершение проверки}: POST \texttt{/entries/<id>/pass} — сохраняет оценку в \texttt{GradeRecord}, меняет статус на \texttt{passed} и пересчитывает позиции.
\end{itemize}

\begin{figure}[ht!]
    \centering
    \includegraphics[width=0.85\textwidth]{screen_queue_detail}
    \caption{Страница активной очереди: список участников с кнопками управления}
    \label{fig:screen_queue}
\end{figure}

Логика очереди реализована в \texttt{queue\_service.py}. Для гарантии FIFO-порядка используется поле \texttt{position}, которое пересчитывается функцией \texttt{normalize\_positions} (листинг~\ref{lst:normalize}).

\begin{lstlisting}[caption={Пересчёт позиций в очереди}, label={lst:normalize}]
def normalize_positions(session_id: int) -> None:
    entries = QueueEntry.query.filter(
        QueueEntry.session_id == session_id,
        QueueEntry.status.in_(["waiting", "checking"])
    ).order_by(QueueEntry.position.asc(),
               QueueEntry.created_at.asc()).all()
    for index, entry in enumerate(entries, start=1):
        entry.position = index
\end{lstlisting}

Функция \texttt{next\_position} (листинг~\ref{lst:next_pos}) вычисляет максимальное значение позиции для активных записей и возвращает его, увеличенное на единицу.

\begin{lstlisting}[caption={Вычисление следующей позиции}, label={lst:next_pos}]
def next_position(session_id: int) -> int:
    max_pos = db.session.query(func.max(QueueEntry.position))
        .filter(QueueEntry.session_id == session_id,
                QueueEntry.status.in_(VISIBLE_QUEUE_STATUSES))
        .scalar()
    return int(max_pos or 0) + 1
\end{lstlisting}

\subsubsection{Разработка CRUD взаимодействия ``Клиент-Сервер'' для сущностей}

Для групп, пользователей и дисциплин реализованы стандартные CRUD-операции:

\begin{itemize}
    \item \textbf{Группы}: POST \texttt{/admin/groups} создаёт новую группу; GET отображает список всех групп.

Все CRUD-операции защищены декораторами \texttt{@login\_required} и \texttt{@role\_required}, а также проверяют корректность входных данных перед записью в базу.

В соответствии с задачей 3.4, реализован функционал загрузки студенческих работ и выгрузки отчётов в формате CSV.

\subsection{Реализация WebSocket-обновления очереди в реальном времени}

Для обеспечения синхронного отображения очереди у всех подключённых пользователей (студентов, преподавателя, администратора) реализован механизм реального времени на базе протокола \textbf{WebSocket} с использованием библиотеки \textit{Flask-Sock}~\cite{flask_docs}. При изменении состояния очереди (добавление, удаление, перемещение, вызов студента) сервер инициирует широковещательную рассылку актуальных данных всем клиентам, подписанным на данную сессию.

Схема работы представлена на листинге~\ref{lst:websocket_broadcast}. Функция \texttt{broadcast\_session\_update} сериализует текущее состояние очереди с помощью \texttt{serialize\_session} (определена в \texttt{queue\_service.py}) и отправляет его во все клиентские очереди, зарегистрированные для указанного \texttt{session\_id}.

\begin{lstlisting}[caption={Широковещательная рассылка обновлений (realtime.py)}, label={lst:websocket_broadcast}]
_clients: dict[int, set[Queue]] = defaultdict(set)
_lock = RLock()

def broadcast(session_id: int, payload: dict[str, Any]) -> None:
    with _lock:
        targets = list(_clients.get(session_id, set()))
    for client_queue in targets:
        client_queue.put(payload)

def broadcast_session_update(session_id: int) -> None:
    session = QueueSession.query.get(session_id)
    if session:
        broadcast(session_id, serialize_session(session))
\end{lstlisting}

WebSocket-маршрут \texttt{/ws/session/<int:session\_id>} (листинг~\ref{lst:ws_route}) принимает подключение от клиента, регистрирует его очередь сообщений в глобальном словаре и отправляет начальный снимок состояния. При каждом обновлении сервер помещает сериализованный JSON в очередь клиента, а клиентский JavaScript получает сообщение и динамически обновляет таблицу очереди без перезагрузки страницы.

\begin{lstlisting}[caption={Обработчик WebSocket-соединения (realtime.py)}, label={lst:ws_route}]
@sock.route("/ws/session/<int:session_id>")
def session_socket(ws, session_id: int):
    if not current_user.is_authenticated:
        ws.close()
        return
    client_queue: Queue = Queue()
    with _lock:
        _clients[session_id].add(client_queue)
    try:
        session = QueueSession.query.get(session_id)
        if session:
            ws.send(json.dumps(serialize_session(session)))
        while True:
            payload = client_queue.get(timeout=25)
            ws.send(json.dumps(payload, ensure_ascii=False, default=str))
    except Exception:
        pass
    finally:
        with _lock:
            _clients[session_id].discard(client_queue)
\end{lstlisting}

На клиентской стороне JavaScript (файл \texttt{static/js/main.js}) при загрузке страницы очереди устанавливает WebSocket-соединение. При получении сообщения типа \texttt{"snapshot"} таблица очереди перерисовывается. Для браузеров, не поддерживающих WebSocket, или в случае разрыва соединения предусмотрен резервный механизм периодического опроса через API-маршрут \texttt{/api/sessions/<id>/snapshot} (листинг~\ref{lst:api_snapshot}).

\begin{lstlisting}[caption={REST-совместимый снимок очереди (api.py)}, label={lst:api_snapshot}]
@api_bp.route("/sessions/<int:session_id>/snapshot")
@login_required
def session_snapshot(session_id: int):
    session = QueueSession.query.get_or_404(session_id)
    return jsonify(serialize_session(session))
\end{lstlisting}

Таким образом, достигнута синхронизация состояния очереди в реальном времени, что критически важно для удобства работы преподавателя и студентов в ходе занятия.
